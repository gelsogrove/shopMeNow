4. Artificial Intelligence & Large Language Model (LLM) Processing

Core to our Service is the use of third-party large language model ("LLM") providers to generate conversational AI responses. We route requests primarily through OpenRouter, which in turn relays to providers such as OpenAI, Anthropic and others.

Safeguards we apply:

  • Data minimisation: we send only the information strictly necessary to generate a contextually relevant response. Full payment data, passwords, government-ID numbers and special-category data (health, biometrics, political opinions) are never included in LLM prompts.
  • No model training: our agreements with LLM providers prohibit the use of customer data to train, fine-tune or improve their public models. Processing is strictly transient (generate response → discard input).
  • Workspace isolation: product catalogues, FAQs and custom prompts injected into requests are scoped to a single workspace and treated as confidential business information.
  • Content filtering: we apply input/output filtering to detect and block harmful, illegal or otherwise policy-violating content before and after LLM processing.
  • Audit trail: every LLM interaction is logged internally with an anonymised reference, allowing traceability without exposing raw user content externally.

The legal basis for AI processing is Art. 6(1)(b) GDPR (necessary for contract performance towards Platform Users) and Art. 6(1)(f) GDPR (legitimate interest for End Customer interactions, balanced against the reasonable expectation of receiving an automated response).

5.1 WhatsApp Message Delivery Providers

For WhatsApp channel integrations, Platform Users can choose from multiple third-party WhatsApp message providers. Each provider has different infrastructure, compliance certifications and geographic coverage. Users select which provider(s) to use during workspace configuration. Supported WhatsApp providers currently include:

  • Meta Platforms (WhatsApp Business API) — Direct integration with Meta's official API, messages routed through Meta's infrastructure (EU / US)
  • WAAPI — European WhatsApp intermediary provider, operates EU data centres (EU)
  • Additional partners — Other certified WhatsApp message delivery providers available in platform settings

Secret Keys & API Credentials: For each selected WhatsApp provider, Platform Users must supply API keys, access tokens or authentication credentials. These secrets are:

  • Encrypted at rest — stored using AES-256 encryption in our database
  • Never logged or transmitted — credentials are never written to logs, error traces or external monitoring systems
  • Access-controlled — only the workspace admin who configured the provider can view or rotate the credentials
  • Rotated on demand — users can regenerate or replace provider credentials at any time via the platform dashboard
  • Used exclusively for provider communication — credentials are sent only to the chosen provider's servers and nowhere else

When a customer message arrives via WhatsApp, eChatbot routes it to the customer's chosen provider for delivery to the carrier. The provider's infrastructure and security practices govern message encryption in transit and at the carrier level. eChatbot does not intercept or store the encryption keys used by WhatsApp carriers.

An up-to-date list of available WhatsApp providers and sub-processors is available in the platform settings. We will notify Platform Users at least 14 days before adding a new sub-processor or provider option, providing an opportunity to migrate to a different provider if desired.

5.2 PayPal Payment Processing & Security

OAuth Token Encryption: When a Platform User connects their PayPal account via OAuth 2.0, we receive an access token and refresh token from PayPal. Both tokens are immediately encrypted using AES-256 encryption before storage in our database. These encrypted tokens are never transmitted to external systems, logged, or displayed in our UI. Only the encrypted ciphertext is stored in the database columns paypalAccessTokenEncrypted and paypalRefreshTokenEncrypted. Tokens are decrypted only at the exact moment they are needed to communicate with PayPal's API, and the plaintext is immediately discarded after use.

Subscription & Recurring Billing Security: For monthly billing, Platform Users set up PayPal subscriptions via PayPal's subscription API. We store only non-sensitive subscription metadata: subscription ID (I-XXXXX format), plan ID (P-XXXXX format), subscription status (ACTIVE/SUSPENDED/CANCELLED), and billing timestamps. We do NOT store PayPal bank account details, card numbers, or other sensitive payment information — PayPal's servers are the sole custodian of those credentials. Billing failure events (e.g., declined payment) are tracked locally with a counter and status field, allowing us to notify users and retry billing without exposing the underlying payment failure reason to untrusted third parties.

9. Security Measures

We take the security of your personal data seriously. Below is a detailed and transparent description of the technical and organisational security controls we have designed, implemented and continuously test within our platform. This section is intentionally specific: we believe users have the right to understand exactly how their data is protected, not just be told it is "safe".

9.1 Authentication & Access Control

  • Password hashing (bcrypt, cost factor 10): Passwords are never stored in plaintext. We use the bcrypt adaptive hashing algorithm with a per-password random salt. Even if our database were fully compromised, recovering plaintext passwords from the stored hashes would be computationally infeasible.
  • Constant-time authentication: Our login flow uses a dummy bcrypt hash comparison when a user account does not exist. This prevents timing-based user-enumeration attacks — an attacker cannot distinguish "user does not exist" from "wrong password" by measuring response time.
  • Two-Factor Authentication (2FA — TOTP with QR Code): We offer and strongly encourage Time-based One-Time Password (TOTP) 2FA for all platform accounts (compatible with Google Authenticator, Authy, Microsoft Authenticator, etc.). During 2FA setup, users scan a QR code containing their encrypted 2FA secret. The QR code is generated securely on-device and is never transmitted to our servers — only the hash of the secret is stored. TOTP codes are verified with a ±1 window to account for clock drift. QR codes are displayed once during setup and not cached or stored.
  • Password reset token hashing (SHA-256): When a user requests a password reset, we generate a cryptographically random 32-byte token. Only the SHA-256 hash of that token is stored in our database. The raw token is sent exclusively to the user's email and is never persisted anywhere on our servers. This means that even with full read access to our database, an attacker cannot exploit reset tokens.
  • 2FA reset token hashing (SHA-256): The same SHA-256 hashing principle applies to 2FA reset links. The raw token leaves our system only via the user's verified email. Database storage contains only the hash.
  • Short-lived reset tokens: All password and 2FA reset tokens have a strict expiry window (typically 1 hour). Used or expired tokens are immediately invalidated and cannot be replayed.
  • Three-layer middleware stack: Every authenticated API endpoint is protected by a mandatory three-layer middleware chain: (1) JWT bearer token validation, (2) session ID header validation (x-session-id), (3) workspace ownership validation (x-workspace-id cross-checked against the JWT claim). All three must pass for a request to proceed.
  • Rate limiting on sensitive endpoints: Login, password reset and 2FA verification endpoints are rate-limited by IP address to mitigate brute-force and credential-stuffing attacks.

9.2 Multi-Tenant Data Isolation (Workspace Security)

  • Mandatory workspace scoping: Every single database query in our system includes a workspaceId filter. It is architecturally impossible for a query to return data belonging to a different tenant. This design eliminates an entire class of cross-tenant data-leakage vulnerabilities.
  • IDOR (Insecure Direct Object Reference) prevention: Before any mutating operation (update, delete) on resources such as certifications, product types or agent configurations, the system verifies that the resource belongs to the requesting workspace. An authenticated but unauthorised tenant cannot manipulate another tenant's data by guessing resource IDs.
  • Privilege escalation prevention (mass-assignment protection): Our API enforces a strict allowlist of fields that can be modified through user-facing endpoints. Fields such as isPlatformAdmin, isDeveloperUser, passwordHash, isVerified, twoFactorEnabled and twoFactorSecret are blocked at the controller layer. Any request containing these fields is rejected with HTTP 403. An additional defense-in-depth strip is applied at the database repository layer, ensuring no privilege-escalation field ever reaches the database even if the controller check were somehow bypassed.
  • Secure time-limited public tokens: Features that require sharing order or customer links publicly (e.g., order tracking pages) use cryptographically signed, time-limited tokens. These tokens encode the workspace, customer and expiry and are validated on every request. They do not require authentication but cannot be forged or reused after expiry.

9.3 File Upload Security

  • MIME-type allowlist: File uploads are validated against a strict allowlist of safe MIME types. Specifically, SVG files and any other file type that could carry embedded executable content are refused. This prevents stored cross-site scripting (XSS) attacks via maliciously crafted uploaded files.
  • Filename sanitisation: All uploaded filenames are stripped of path traversal sequences (../, ./), non-ASCII characters, whitespace and potentially dangerous characters before storage. Only alphanumeric characters, hyphens, underscores and a single dot (for the extension) are permitted.
  • Path traversal prevention: File serving endpoints validate that the requested resource path begins with the requesting workspace's own prefix. It is impossible for a user to retrieve files belonging to another workspace by manipulating the file path.
  • Maximum file size enforcement: Upload middleware enforces file-size limits to prevent denial-of-service via large-payload flood attacks.

9.4 Concurrency & Data Integrity

  • Atomic stock management: Product stock decrements are performed as atomic database operations with a guard condition (stock ≥ quantity). Under high concurrency — multiple simultaneous orders for the same product — only the operations that can be satisfied will succeed. This prevents overselling (stock going negative) without the need for application-level locking.
  • Transaction-safe session creation: Customer chat sessions are created inside database transactions with a unique constraint on (customerId, status=active). Concurrent session-creation requests for the same customer are safe: duplicate-key errors are caught and the existing session is returned, preventing ghost sessions and data duplication.
  • Order code collision handling: Order codes are generated with a retry loop that handles unique-constraint violations (Prisma P2002). In the rare event of a collision under high concurrency, a letter suffix is appended and the operation retries up to five times.
  • Per-customer message processing locks: Incoming messages from the same customer are processed sequentially using an in-memory per-customer lock. This prevents race conditions where two near-simultaneous messages from the same customer could result in duplicate AI responses or inconsistent cart state.

9.5 Infrastructure & Transport Security

  • Encryption in transit (TLS 1.2+): All communications between clients and our servers — web application, API, WebSocket channels — are encrypted using TLS 1.2 or higher. Unencrypted HTTP connections are redirected to HTTPS.
  • Encryption at rest: Our managed PostgreSQL database and file storage use AES-256 encryption at rest. Database backups are also encrypted.
  • Credential protection: Administrative credentials (database passwords, API keys, third-party secrets) are managed exclusively via environment variables and secret management services. They are never hard-coded in source code, never logged, and never exposed in process listings. Database backup operations pass the database password via a dedicated environment variable (PGPASSWORD), not via command-line arguments, so it cannot be read from the operating system process table (ps aux).
  • Database connection pool management: Our application uses a shared, long-lived Prisma client with a managed connection pool. Connections are never explicitly disconnected within business logic, preventing premature pool exhaustion and connection leaks under load.
  • Sensitive data exclusion from logs: API keys, tokens, password hashes and customer PII are explicitly excluded from application logs. Where token presence must be confirmed (e.g., for debugging), only the first 8 characters of a token are logged, never the full value.

9.6 What We Do NOT Do With Your Data

  • We do not sell personal data — to any party, under any circumstances.
  • We do not share personal data for advertising — we have no advertising partnerships and do not install third-party advertising trackers.
  • We do not use customer conversation data to train AI models — our agreements with LLM providers include zero-retention clauses. Customer messages are not used to improve public AI models.
