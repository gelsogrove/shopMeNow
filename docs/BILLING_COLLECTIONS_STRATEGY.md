# Billing & Collections Strategy

**Last updated**: 2026-03-11
**Author**: Andrea Gelsomino

---

## Overview

eChatbot uses a **manual-trigger PayPal billing model**: invoices are auto-generated by the scheduler, but payments are collected manually by the admin through the Backoffice → Collections page.

The system is **two-tier**:
1. **Subscription fee** — Fixed monthly cost (€19 / €49 / €99) based on plan
2. **Credit usage** — Pay-as-you-go wallet (€0.10/message, €1.00/push, €1.50/order)

---

## Invoice Lifecycle

```
DRAFT ──▶ PENDING ──▶ PAID
                  ↘ FAILED (on PayPal error, retryable)
                  ↘ CANCELLED (admin manually cancelled)
```

| Status | When | Admin Can Download PDF? | User Can Download PDF? |
|--------|------|------------------------|------------------------|
| DRAFT | Current month in progress | ✅ Yes (preview) | ❌ No |
| PENDING | Month closed, awaiting payment | ✅ Yes (preview) | ❌ No |
| PAID | Payment confirmed by PayPal | ✅ Yes | ✅ Yes |
| FAILED | PayPal charge failed | ✅ Yes (preview) | ❌ No |
| CANCELLED | Manually voided | ✅ Yes | ❌ No |

> **PDF Generation**: On-demand using `pdfkit`. No pre-generated files stored. Admin can download at any status; user can download only PAID invoices.

## Billing Flow (End-to-End)

```
1st of month (23:30 UTC)
     │
     ▼
[Scheduler: monthly-billing.job.ts]
     │  Creates PENDING invoice for each active owner
     │  Calculates: subscription_fee + credit_debt
     │  Does NOT call PayPal
     │
     ▼
[Admin: Backoffice → Collections → Previous Month]
     │  Reviews unpaid invoices
     │  Clicks "Process Payment"
     │
     ▼
[Backend: POST /admin/invoices/:id/paypal/process-payment]
     │  Validates PayPal subscription is ACTIVE
     │  Calls PayPal: POST /v1/billing/subscriptions/{id}/capture
     │  Creates PayPalTransaction record (PENDING state)
     │
     ▼
[PayPal Webhook: PAYMENT.SALE.COMPLETED or PAYMENT.SALE.DENIED]
     │  SUCCESS → Invoice.status = PAID, generates invoice number
     │  FAILED  → PayPalTransaction.status = FAILED, invoice stays PENDING
```

## Invoice Calculation Formula

```
subscriptionAmount  = monthly fee from PlanConfiguration table (NOT hardcoded)
creditUsage         = SUM of debit transactions in period (MESSAGE, ORDER, PUSH)
creditDebt          = MAX(0, -user.creditBalance)  ← negative balance carried forward
recharges           = SUM of credit top-ups in period
adjustments         = manual admin adjustments (can be negative)

subtotal            = subscriptionAmount + creditDebt + recharges + adjustments
tax (22% IVA)       = subtotal × 0.22
total               = subtotal + tax
```

**No hardcoded values** — all plan fees and usage rates come from the `PlanConfiguration` database table.

## Collections Page — Tab Semantics

### Current Month (`/collections` → "Current Month")
- **What**: Live preview of this month's accumulating charges
- **Invoice Status**: DRAFT
- **Action**: View only, no payment processing yet
- **Download**: ✅ Admin can download draft invoice PDF

### Previous Month ("Previous Month" tab)
- **What**: Invoices from past months with status DRAFT or PENDING — i.e., **not yet paid**
- **Invoice Status**: DRAFT or PENDING
- **Action**: Process Payment (triggers PayPal capture)
- **Download**: ✅ Admin can always download (regardless of status)
- **Note**: This is the main collection workstation — where admin charges users

### Failed ("Failed" tab)
- **What**: Invoices where PayPal charge returned an error
- **Invoice Status**: FAILED
- **Action**: Retry Payment, Move to History (archive), Move to Trash (cancel)
- **Download**: ✅ Admin can download (added 2026-03-11)
- **Common errors**:
  - `"The requested action could not be performed, semantically incorrect, or failed business validation"` → PayPal sandbox limitation or subscription not ACTIVE
  - `env:sandbox` suffix → user is developer/admin, charged in sandbox mode

### History ("History" tab)
- **What**: PAID invoices only (filtered client-side)
- **Invoice Status**: PAID
- **Action**: Download PDF, Add Credit Note
- **Download**: ✅ Full invoice + credit notes
- **Note**: The `loadHistory()` API returns all statuses, but the frontend filters to PAID only (line 349 of CollectionsPage.tsx)

### Transactions ("Transactions" tab)
- **What**: Successful PayPal payment transactions
- **Data Source**: `PayPalTransaction` table filtered by `status=SUCCESS`
- **Download**: ✅ Each row has a PDF download button linked to the invoice (added 2026-03-11)
- **Note**: Shows only SUCCESS transactions — FAILED ones are visible in the Failed tab

## Scheduler Status

### Monthly Billing Job
- **File**: `apps/scheduler/src/jobs/monthly-billing.job.ts`
- **Cron**: `30 23 1 * *` → runs on **1st of every month at 23:30 UTC**
- **Enable/Disable**: Backoffice → Schedulers page (toggle `isActive` flag in DB)
- **Check**: `JobRunner` queries `isActive` from DB before each run — if `false`, job is skipped
- **What it does**:
  1. Fetches all active workspace owners
  2. Skips PAUSED users (no billing)
  3. Skips FREE_TRIAL users (or pauses them if trial expired)
  4. Applies pending plan downgrades
  5. Calculates total charge (subscription + credit debt)
  6. Creates/upserts PENDING invoice
  7. Updates `nextBillingDate` to 1st of next month

### Rate Limiting Protection
The `paypal-billing.service.ts` has an **in-memory rate limiter** (60 seconds between attempts per invoice). This prevents rapid double-charges but is reset on server restart.

> ⚠️ **Known Issue**: Rate limiter is in-memory only — if the backend restarts between two rapid clicks, the limit is bypassed. See also: `docs/SCHEDULER_ISSUES_ANALYSIS.md` for race condition notes.

## Production Readiness Audit (2026-03-11)

### Will it work in production? YES — with conditions.

The `/v1/billing/subscriptions/{id}/capture` with `capture_type: OUTSTANDING_BALANCE` is the correct PayPal MIB endpoint. The custom `amount` in the body IS used by PayPal to charge that specific amount on the subscription.

**Required conditions for production to work**:
1. User has connected PayPal LIVE account (OAuth flow completed)
2. `BILLING.SUBSCRIPTION.ACTIVATED` webhook fired → subscription is `ACTIVE` in DB
3. First $1 billing cycle has been processed (PayPal requires at least one cycle)
4. `PAYPAL_WEBHOOK_ID_LIVE` is configured in env
5. Webhook endpoint is reachable from PayPal

**Why sandbox fails**: PayPal sandbox requires you to manually simulate the first billing cycle via the developer dashboard. Without this, the outstanding balance capture is rejected with "semantically incorrect". This is a sandbox limitation, NOT a production bug.

### Bugs Fixed (2026-03-11)

| Bug | Severity | Fix Applied |
|-----|---------|-------------|
| Rate limiter declared but never used → double charges possible | 🔴 Critical | Added `processingInvoices.get/set` check in `processPayment()` |
| Invoice matching fell back silently to oldest invoice | 🟡 Medium | Added warning log when amount doesn't match; fallback still applies but is now traceable |
| `handleMockPayment` didn't update `currentRows` → button stayed active | 🔴 Critical | Fixed in CollectionsPage.tsx |
| Transactions tab showed FAILED | 🟡 Medium | Fixed to filter SUCCESS only |
| No download in Failed tab | 🟡 Medium | Added Download button |

### Remaining Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Rate limiter is in-memory → resets on backend restart | If server restarts between two rapid clicks, 60s window is lost | Accept for now; low probability |
| Webhook dependency → invoice stays PENDING if webhook fails | Admin sees PENDING even after PayPal charged | Add "Mark as Paid Manually" button as fallback |
| Invoice matching by amount (fallback to oldest) | Could mark wrong invoice PAID if user has multiple pending with same amount | Acceptable for current scale; improve with note-based matching later |
| In-memory plan cache lost on restart | Duplicate products/plans created in PayPal | Move planId to database or use env var `PAYPAL_PLAN_ID_LIVE` |

## What Works ✅

- Monthly scheduler creates PENDING invoices on 1st of month
- Admin can process payment from Previous Month tab
- PayPal sandbox/live environment auto-detection per user
- PayPal webhook confirms payment → invoice marked PAID
- PDF generation on-demand for any invoice status
- Rate limiter (60s) now active — prevents rapid double-clicks
- Collections UI updates immediately after payment attempt
- Transactions tab shows only SUCCESS
- Download button available in: Previous Month, Failed, History, Transactions tabs

## What Needs Attention ⚠️

- **Sandbox testing**: Must simulate first billing cycle in PayPal Developer Dashboard before testing captures
- **Webhook must be configured**: `PAYPAL_WEBHOOK_ID_LIVE` in env + PayPal → Webhooks → `BILLING.SUBSCRIPTION.PAYMENT.SUCCESS` pointing to `/api/v1/paypal/webhook`
- **History tab**: `loadHistory()` returns all statuses but FE filters to PAID only (line 349 CollectionsPage.tsx)
- **Race condition in scheduler**: See `docs/SCHEDULER_ISSUES_ANALYSIS.md`
