# Implementation Plan: Guided Progressive Product Search

**Branch**: `123-guided-product-search` | **Date**: 2025-01-12 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/123-guided-product-search/spec.md`

## Summary

Implement guided product search with dynamic LLM-driven grouping that progressively filters product catalog from broad categories down to max 3 products, enabling direct cart addition with quantity support. Core innovation: LLM analyzes customer query intent and chooses optimal grouping strategy (categories, certifications, price, use-case) rather than fixed category trees. Implementation is **90% prompt engineering** with ProductSearchAgent prompt rewrite, plus minor calling function enhancement for quantity parameter. No flow changes: Router → ProductSearchAgent → Router → CartManagementAgent preserved.

## Technical Context

**Language/Version**: TypeScript 5.x / Node.js 18+  
**Primary Dependencies**: OpenRouter (GPT-4o-mini), Prisma ORM, Express.js  
**Storage**: PostgreSQL (products with certifications: isDOP, isBio, isHalal, isIntegrale)  
**Testing**: Jest (unit tests for CF, integration tests for agent flows)  
**Target Platform**: Node.js backend + WhatsApp chat interface  
**Project Type**: Web application (backend-focused, no frontend changes)  
**Performance Goals**: <3 seconds for initial grouping, <5 seconds total search completion  
**Constraints**: GPT-4o-mini token limit (128k context, aim <50k with {{PRODUCTS}}), max 8 function calling iterations  
**Scale/Scope**: Workspace-isolated (multi-tenant), 50-500 products per workspace typical, seed database enhancement needed

#### Prompt Structure (product-search-agent.md)

**Section 1: Role & Context**

- Specialist agent for product discovery
- Access to workspace product catalog via {{PRODUCTS}}
- Goal: Help customer find right product through progressive filtering

**Section 2: Dynamic Grouping Logic**

- Analyze customer query intent (category, dietary, price, use-case, certification)
- Choose grouping strategy based on query context
- Examples (few-shot learning):
  - "formaggi" → Group by categories (Freschi, Stagionati, Molle, DOP)
  - "prodotti halal" → Group by certification (Halal formaggi, Halal salumi)
  - "regali natale" → Group by price (€10-€20, €20-€40, €40+)
  - "aperitivo" → Group by use-case (Formaggi da tavola, Salumi affettati, Vini)

**Section 3: Progressive Filtering Rules**

- Show groups when >3 products match
- Continue filtering until ≤3 products remain
- Skip grouping if only 1 product matches (show details directly)

**Section 4: Product Display Format**

- Final list (≤3 products): Name, Price (original → discounted), Certifications, Origin
- Detailed view (after selection): Full description, weight, stock status

**Section 5: Cart Integration Instructions**

- Recognize "sì lo voglio", "aggiungi al carrello" as cart intent
- Extract quantity: "ne voglio 5" → qty=5, default qty=1
- Delegate to Router with "ADD_TO_CART: productId, quantity"

**Section 6: Edge Cases Handling**

- Empty results: Suggest alternatives
- Out of stock: Offer similar products
- Ambiguous quantity: Ask clarification

## Risk Mitigation Plan

### Risk 1: Token Limit Exceeded (HIGH)

**Scenario**: Workspace with 1000+ products causes {{PRODUCTS}} variable to exceed GPT-4o-mini 128k context limit.

**Detection**:

- Log {{PRODUCTS}} token count in PromptProcessorService
- Alert when >50k tokens (50% of safe limit)

**Mitigation**:

1. **Short-term**: Add product count check in ProductSearchAgent
   - If >500 products, use category-based pre-filtering before LLM
   - Example: Customer says "formaggi" → Filter products by category="Formaggi" → Send only 50-100 products to LLM
2. **Long-term**: Implement RAG-based product search (Feature 124+)
   - Embed product descriptions in vector database
   - Semantic search retrieves top 50 relevant products
   - Send only relevant subset to LLM

**Fallback**: If token limit still exceeded, default to simple category grouping (no dynamic LLM grouping).

### Risk 2: LLM Grouping Quality (MEDIUM)

**Scenario**: LLM produces inconsistent or unhelpful groupings ("Group 1", "Group 2" instead of meaningful names).

**Detection**:

- Log all grouping decisions to analytics table
- Review logs weekly for quality patterns

**Mitigation**:

1. **Few-shot learning**: Include 10-15 high-quality grouping examples in prompt
2. **Explicit constraints**: "Group names MUST be customer-friendly Italian (e.g., 'Formaggi freschi', NOT 'Fresh cheeses')"
3. **Fallback logic**: If LLM returns generic group names, default to category-based grouping
4. **A/B testing**: Compare dynamic grouping vs. fixed categories, measure conversion rate

**Success Metric**: 90% of groupings use meaningful Italian names (measured via regex: `/^[A-Z][a-z]+ [a-z]+$/`).

### Risk 3: Quantity Extraction Ambiguity (LOW)

**Scenario**: Customer says "ne voglio alcuni", "tipo 3-4", "qualche pezzo" → unclear quantity.

**Detection**: Log all quantity extraction attempts with customer message.

**Mitigation**:

1. **Prompt instruction**: "Extract ONLY explicit numbers (1, 2, 5, 10). If ambiguous, ask: 'Quanti esattamente?'"
2. **Regex validation**: `quantity = message.match(/\b(\d+)\b/)` - extract first number only
3. **Default fallback**: If extraction fails after clarification attempt, default to qty=1 with logged warning

**Success Metric**: <5% of quantity extractions require clarification (measured via logs).
